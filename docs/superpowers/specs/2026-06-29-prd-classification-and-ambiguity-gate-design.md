# PRD 모호성 제거 + 계층 분류 강화 — 설계

> 출처: 사용자 피드백(2026-06-29). harness-authoring 의 greenfield PRD 생성이
> 너무 두루뭉실하다 → 모호함은 분석 미완의 신호이므로 PRD 작성 시 사용자에게
> 모두 묻고, 기능/비기능 외에 도메인·사용자권한 등 2~3차 하위분류를 둔다.

## 문제

현재 harness-authoring 의 PRD 생성에는 두 가지 약점이 있다.

1. **질문 게이트가 약하다** — 룰 8-1 은 PRD 필수 슬롯의 *공백만* `AskUserQuestion`
   으로 묻는다. 값이 있되 **모호한**(측정불가·복수해석·범위불분명) 항목은 그대로
   통과해 두루뭉실한 PRD 가 나온다.
2. **분류가 평면적이다** — `prd.template.md` 는 기능/비기능 2분류뿐. 도메인·사용자
   권한 같은 하위 축이 없어 요구사항이 한 덩어리로 뭉친다.

## 결정 (사용자 확정)

- 수정 대상: **harness-authoring PRD 시스템**(3개 SSOT 일관 수정).
- 질문 범위: **공백 + 모호 모두** 질문, 측정가능·단일해석이 될 때까지.
- 분류 구조: **고정 스키마**(템플릿에 하위분류 섹션을 명시). 단, 적용 안 되는 축은
  삭제가 아니라 `해당 없음 — 사유`로 남겨 누락과 구분한다(룰 9-2 "관심사 실재할 때만
  emit" 과 정렬 — 고정 골격에서 출발하되 근거 없는 축은 비우지 않고 사유를 남김).

## 변경

### 1. prd.template.md — 평면 → 계층 고정 스키마

- **§3 사용자 / 시나리오**: 사용자 권한(role) 분류를 명시(§4 권한 축과 연결).
- **§4 기능 요구사항**: `도메인(1차) > 사용자권한/하위영역(2차) > 개별 FR(3차)`.
  각 FR = `ID · 설명 · 우선순위 · 수용 기준(측정가능)`. 모호/미상은 `확인 필요`.
- **§5 비기능 요구사항**: ISO/IEC 25010 정렬 고정 하위축 — 성능 · 보안 · 가용성 ·
  확장성 · 접근성 · 유지보수성 · 호환성. 각 축은 **정량 기준** 또는 `해당 없음 — 사유`.

### 2. harness-rules.md 룰 8-1 — 질문 게이트 강화

- "공백만 묻는다" → **"공백 + 모호(측정불가·복수해석·범위불분명) 모두 묻는다,
  측정가능·단일해석이 될 때까지"**.
- 분류축(도메인·권한 등)·깊이(2~3차)를 scope 명확화 대상에 포함 — 어떤 축이
  적용되는지 질문.
- Karpathy(rule 4)·추측 금지·`확인 필요` fallback 유지.

### 3. tech-doc-guide.md PRD 절 — 작성법 갱신

- 새 계층 구조 + 수용 기준 의무 + 모호 항목 질문 규율 반영.

## 비목표 (YAGNI)

- brownfield PRD(애초에 생성 안 함).
- 분류축을 프로젝트마다 자유 선택(고정 스키마 결정으로 기각).
- `.py` 변경 — 본 변경은 .md 3개로 한정(Fast 티어).

## 정합성 불변식

- "한 사실 한 곳": 룰=강제 directive, 지침=작성법, 템플릿=골격. 분류축 목록의 SSOT 는
  템플릿이고 룰·지침은 이를 가리킨다(중복 정의 금지).
