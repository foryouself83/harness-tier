# 메모리 정리·SSOT 이관 규율 (SSOT)

`/harness-insight` Step 5 가 따르는 **권위 규율**. 누적된 프로젝트 메모리를 검토해
(1) 무효·중복은 **삭제(prune)**, (2) 지속적 지식은 프로젝트의 `.claude/rules/`·`docs/`
로 **승격(promote)**, (3) 크로스 프로젝트 습관은 메모리에 **유지(keep)** 한다.
**파괴적 동작(삭제·이관)은 사용자 승인 후에만.** 규율 변경은 이 파일에서만 한다.

프로젝트 비종속 — 메모리 경로·문서 형식·언어를 **대상 프로젝트에서 도출**한다
(하드코딩·모델 지식으로 지어내지 않는다).

---

## 1. 데이터 출처

- 메모리 위치: `<project_dir>/memory/` — `<project_dir>` 은 Step 2 스크립트가
  `scanning <dir>` 로 출력한 **베이스 프로젝트 디렉터리**(cwd 슬러그; worktree 형제 제외).
  즉 `~/.claude/projects/<slug>/memory/`.
- 구성: `MEMORY.md`(인덱스 — 한 줄 = 한 메모리) + 각 `*.md`(frontmatter + 본문).
- frontmatter `metadata.type`: `user` | `feedback` | `project` | `reference`.
- 메모리 디렉터리가 없거나 비어 있으면 Step 5 를 **건너뛴다**(정리할 것 없음 — 보고만).
- 로컬 파일 읽기뿐 — 메모리 내용은 기기 밖으로 나가지 않는다.

## 2. 분류 (각 메모리 → 하나)

- **삭제(prune)** — 무효/오답, 상위 문서로 대체됨, 특정 과거 대화에만 유효했던 것,
  **repo 가 이미 기록**(코드·git 이력·CLAUDE.md·기존 문서)한 것, 승격 완료 후의 원본.
- **승격(promote)** — 버전관리·팀 공유돼야 할 지속적 `project`/`reference` 지식
  (아키텍처·설계 결정·운영 함정·컨벤션·외부 리소스). 대상 선택은 §3, 작성은 §4.
- **유지(keep)** — `user`/`feedback` 중 **크로스 프로젝트·개인 습관/선호**
  (특정 repo 문서에 안 맞는 것). 건드리지 않는다.

  > 판단 기준: **"이 저장소를 다루는 지식인가"**(→승격) vs
  > **"이 사람과 일하는 방식인가"**(→유지). 후자는 메모리가 제 위치다.

## 3. 승격 대상: rules vs docs

프로젝트에 **실재하는 관례를 우선**한다(Step 3 에서 확인한 것만).

- **`.claude/rules/`**(또는 프로젝트의 `rules/`) — 모델이 **매 작업에서 지켜야 할
  항상-적용 규율**: 코딩 표준, 커밋 규율, 금지사항, 정책.
- **`docs/`** — **참조성 지식**: 아키텍처·설계 결정·운영 런북·함정·트러블슈팅·외부 링크.
- 둘 중 하나만 있으면 그쪽. 둘 다 있으면 성격으로 나눈다(규율→rules, 참조→docs).
- 둘 다 없으면 `docs/` 신설 또는 사용자에게 위치를 확인한다.

## 4. 승격 작성 규율 (대상 프로젝트 기준)

1. **형식·언어 확인** — 대상 rules/docs 의 **기존 문서를 먼저 읽어** 헤더 스타일·
   언어(한/영)·톤·경로 컨벤션을 맞춘다. 모델 지식으로 형식을 지어내지 않는다.
2. **핵심 유지·간결** — 메모리의 Why/How 를 **핵심만** 남겨 압축한다. 장황한 사건
   서술은 규칙·근거 한두 줄로 줄인다.
3. **SSOT 준수(중복 금지)** — 같은 사실이 이미 기존 문서에 있으면 **새 파일을 만들지
   말고 병합/링크**한다. 없을 때만 신규 파일. 한 값(버전·경로·정책)은 한 곳에만 두고
   나머지는 링크로 defer.
4. **point-in-time 검증** — 메모리가 `file:line`·심볼·플래그를 인용하면 **현재 코드로
   존재를 확인한 뒤** 승격한다(오래된 인용을 사실로 박제 금지). 확인 불가한 부분은 뺀다.

## 5. 적용 (사용자 승인 게이트)

1. **제안 표를 대화로 제시** — 컬럼: `메모리 | 분류(삭제/승격/유지) | 대상 SSOT | 근거`.
   유지 항목도 표에 "유지"로 넣어 왜 승격 안 하는지 밝힌다.
2. **승인 후에만 적용** — `AskUserQuestion` 등으로 확인받은 뒤:
   - (a) 승격: 대상 rules/docs 에 §4 규율로 쓴다(기존 문서 **병합 우선**).
   - (b) 삭제: prune 항목 + 승격 완료 원본 메모리 파일 제거.
   - (c) `MEMORY.md` 인덱스 갱신 — 삭제·승격된 줄 제거, **유지 항목만** 남긴다.
3. 승인 없이 삭제·이동·덮어쓰기 금지. **부분 승인**(일부만 적용)도 지원한다.

---

## Critical

1. **승인 게이트 필수** — 메모리 삭제·이관은 파괴적. 제안 → 승인 → 적용 순서를 어기지 않는다.
2. **유지 항목 불가침** — keep 분류는 삭제·이관하지 않는다.
3. **SSOT 중복 금지** — 기존 문서에 있으면 병합/링크, 신규 파일 남발 금지.
4. **대상 경로** — 쓰기는 **호스트 프로젝트**의 rules/docs 와 `<project_dir>/memory/` 뿐.
   플러그인 디렉터리에 쓰지 않는다.
5. **로컬 처리만** — 메모리 내용은 기기 밖으로 나가지 않는다.
