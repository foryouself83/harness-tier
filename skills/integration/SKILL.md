# integration

**Manually runs the project's integration tests**, or, when there is no automation, **collects scenarios from a human**.
This is a manual skill, not a gate â€” it is not tied to the `/flow` gate.

> **Positioning**: This skill **deterministically runs existing Playwright cases**. If it is web but has
> no cases, it uses `playwright-scaffold` to generate only a **main-screen smoke** (a deterministic
> "does the app come up?" check) and run it (it does not generate arbitrary scenarios); if it is not
> web, it is handled human-in-the-loop.
> Web-frontend detection is a heuristic, not a definitive SSOT â€” for details, see
> [`references/web-playwright.md`](references/web-playwright.md).

## 2. Web-Frontend / Electron / Non-web Detection (heuristic)

> For the full list of detection signals and its limitations, see â†’ [`references/web-playwright.md`](references/web-playwright.md).

Check **Electron first** â€” it is a distinct verdict, not folded into Non-web:

```bash
grep '"electron"' package.json 2>/dev/null
```

If not Electron, check `package.json`'s `dependencies`/`devDependencies` against a web-framework allowlist:

```bash
# Check for web-framework dependencies
grep -E '"(react|vue|next|nuxt|svelte|@angular/core|solid-js|astro)"' package.json 2>/dev/null
```

Also check the supporting signals:

```bash
# Supporting signals: vite.config, index.html, public/
ls vite.config.* index.html public/ 2>/dev/null
```

Check the **other non-web signals** (if present, classify as non-web):

```bash
# CLI: bin field, RN: react-native/metro.config.js, Flutter: pubspec.yaml, Go: go.mod/main.go
grep '"bin"' package.json 2>/dev/null
grep '"react-native"' package.json 2>/dev/null
ls metro.config.js pubspec.yaml go.mod main.go 2>/dev/null
```

| Verdict | Condition |
|---|---|
| **Electron** | An `"electron"` dependency exists (checked **before** any other signal) |
| **Web** | No Electron signal + an allowlist dependency exists + no other non-web signal |
| **Non-web** | No Electron signal + a non-web signal exists (CLI/RN/Flutter/Go/etc.), or no allowlist match |

> For the full **Electron** procedure (renderer automation + main-process human-in-the-loop), see
> [`references/electron.md`](references/electron.md) â€” the single source of truth for this exception.

## 3. If Web â€” Run Existing Playwright Cases

> For details on Playwright config, the testDir/testMatch defaults, and reporter options, see
> â†’ [`references/web-playwright.md`](references/web-playwright.md).

### 3.1 Parse playwright.config.*

```bash
# Detect the config file
ls playwright.config.ts playwright.config.js playwright.config.mjs playwright.config.cjs 2>/dev/null | head -1
```

Read `testDir` and `testMatch` from the config file. The defaults are:
- `testDir`: `./tests`
- `testMatch`: `**/*.@(spec|test).?(c|m)[jt]s?(x)`

### 3.2 Discover Existing Cases

```bash
# Matches the testMatch default exactly: **/*.@(spec|test).?(c|m)[jt]s?(x)
find ./tests -regextype posix-extended -regex '.*\.(spec|test)\.(c|m)?[jt]sx?' 2>/dev/null | wc -l
```

**If there are zero cases**, invoke the `playwright-scaffold` skill to **generate a main-screen smoke and run it immediately**
(not an arbitrary scenario, but a deterministic "does the app come up?" check):

1. Invoke `playwright-scaffold` via the `Skill` tool â†’ confirm baseURL (codebase scan + user confirmation)
   â†’ idempotently generate `<testDir>/main.smoke.spec.*`. (If Playwright is not installed, the scaffold guides installation.)
2. Run the generated smoke with `--reporter=json` per Â§3.3 and report PASS/FAIL per Â§3.4.
3. In the report, state **"this is a starter smoke, so extend it with real cases"** (`npx playwright codegen <baseURL>`).

> The runner is `npx playwright test` (deterministic, SSOT). Use the session's playwright MCP only as an
> auxiliary path for exploration when cases are absent (it is not the regression SSOT). MCP reference:
> the "playwright MCP" section of [`references/web-playwright.md`](references/web-playwright.md).

### 3.3 Deterministic Run

If cases exist, run deterministically with `--reporter=json` (+junit). `--reporter=json` emits to
**stdout** by default, so to capture it to a file, set `PLAYWRIGHT_JSON_OUTPUT_NAME` (otherwise the
file parsing in Â§3.4 breaks with ENOENT):

```bash
PLAYWRIGHT_JSON_OUTPUT_NAME=results.json \
PLAYWRIGHT_JUNIT_OUTPUT_NAME=results.xml \
  npx playwright test --reporter=json,junit
```

### 3.4 Parse Results and Report PASS/FAIL

Parse the `results.json` specified in Â§3.3, defensively (a crashed or incomplete Playwright run must still
produce a clear FAIL report, not an uncaught exception):

```bash
node -e "
  const fs = require('fs');
  let r;
  try { r = JSON.parse(fs.readFileSync('results.json', 'utf8')); }
  catch (e) { console.log('FAIL: results.json missing or invalid (' + e.message + ')'); process.exit(1); }
  const s = r && r.stats;
  if (!s || typeof s.unexpected !== 'number') {
    console.log('FAIL: results.json has no stats â€” the Playwright run likely crashed before completing');
    process.exit(1);
  }
  console.log('PASS:', s.expected, '/ FAIL:', s.unexpected, '/ SKIP:', s.skipped);
  process.exit(s.unexpected > 0 ? 1 : 0);
"
```

**Report format**:

```
## Integration Test Results â€” <date>

### Playwright Run Results
| Suite | Passed | Failed | Skipped | Verdict |
|---|---|---|---|---|
| tests/auth.spec.ts | 5 | 0 | 0 | PASS |
| tests/checkout.spec.ts | 3 | 1 | 0 | FAIL |

**Overall verdict**: FAIL (1 failure)
Failed case: tests/checkout.spec.ts â€ş navigate to the confirmation page after checkout completes
```

## 5. If Non-Web â€” human-in-the-loop

> For per-type non-web signals and reference OSS details, see â†’ [`references/non-web.md`](references/non-web.md).

Do not enforce automated integration testing. Determine the specific non-web type from the Â§2 signal that
matched (CLI / React Native / Flutter / Go / generic â€” do not fall back to a generic list if a specific
signal matched), then collect scenarios and pass criteria via `AskUserQuestion` using that type:

```
This project was detected as non-web (<type>).
No integration-test automation tool is enforced.

Please provide the following:
1. The core scenarios to verify (e.g., "look up data after user login")
2. The pass criteria for each scenario (e.g., "response code 200, data included")
3. Any test tool you are currently using, if any.
```

Based on the collected scenarios, write a manual verification checklist and point to applicable OSS
(Newman/Maestro/Appium â€” all Apache-2.0, not automatically enforced). For details, see
[`references/non-web.md`](references/non-web.md).
