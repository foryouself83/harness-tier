# integration

**Manually runs the project's integration tests**, or, when there is no automation, **collects scenarios from a human**.
This is a manual skill, not a gate — it is not tied to the `/flow` gate.

> **Positioning**: This skill **deterministically runs existing Playwright cases**. If it is web but has
> no cases, it uses `playwright-scaffold` to generate only a **main-screen smoke** (a deterministic
> "does the app come up?" check) and run it (it does not generate arbitrary scenarios); if it is not
> web, it is handled human-in-the-loop.
> Web-frontend detection is a heuristic, not a definitive SSOT — for details, see
> [`references/web-playwright.md`](references/web-playwright.md).

---

## 1. Consume Host Documentation First

```bash
# Check for integration documentation generated by harness-researcher (Phase 4; consume first if present)
ls docs/verification/integration.md 2>/dev/null && cat docs/verification/integration.md
```

- **If `docs/verification/integration.md` exists**: follow the file's per-stack integration strategy, tools, and run commands.
- **If `docs/verification/integration.md` is absent**: fall back to `references/web-playwright.md` and `references/non-web.md`.

## 2. Web-Frontend Detection (heuristic)

> For the full list of detection signals and its limitations, see → [`references/web-playwright.md`](references/web-playwright.md).

Check `package.json`'s `dependencies`/`devDependencies` against a web-framework allowlist:

```bash
# Check for web-framework dependencies
grep -E '"(react|vue|next|nuxt|svelte|@angular/core|solid-js|astro)"' package.json 2>/dev/null
```

Also check the supporting signals:

```bash
# Supporting signals: vite.config, index.html, public/
ls vite.config.* index.html public/ 2>/dev/null
```

Check the **non-web signals** (if present, classify as non-web):

```bash
# CLI: bin field, RN: react-native/metro.config.js, Flutter: pubspec.yaml, Electron: electron
grep '"bin"' package.json 2>/dev/null
grep '"react-native"\|"electron"' package.json 2>/dev/null
ls metro.config.js pubspec.yaml 2>/dev/null
```

| Verdict | Condition |
|---|---|
| **Web** | An allowlist dependency exists + no non-web signal |
| **Non-web** | A non-web signal exists, or no allowlist match |
| **Electron** | `"electron"` dependency — Chromium-based internally, so partial automation is possible (web-playwright §1.3) |

> For the **Electron exception** details (partial renderer automation / the main process is human-in-the-loop), see
> [`references/web-playwright.md`](references/web-playwright.md) §1.3 — keep it in one place to avoid duplicate definitions.

## 3. If Web — Run Existing Playwright Cases

> For details on Playwright config, the testDir/testMatch defaults, and reporter options, see
> → [`references/web-playwright.md`](references/web-playwright.md).

### 3.1 Parse playwright.config.*

```bash
# Detect the config file
ls playwright.config.ts playwright.config.js playwright.config.mjs playwright.config.cjs 2>/dev/null | head -1
```

Read `testDir` and `testMatch` from the config file. The defaults are:
- `testDir`: `./tests`
- `testMatch`: `**/*.@(spec|test).?(c|m)[jt]s?(x)`

### 3.2 Discover Existing Cases

```bash
# Count case files in testDir (default ./tests)
find ./tests -name "*.spec.ts" -o -name "*.spec.js" -o -name "*.test.ts" -o -name "*.test.js" 2>/dev/null | wc -l
```

**If there are zero cases**, invoke the `playwright-scaffold` skill to **generate a main-screen smoke and run it immediately**
(not an arbitrary scenario, but a deterministic "does the app come up?" check):

1. Invoke `playwright-scaffold` via the `Skill` tool → confirm baseURL (codebase scan + user confirmation)
   → idempotently generate `<testDir>/main.smoke.spec.*`. (If Playwright is not installed, the scaffold guides installation.)
2. Run the generated smoke with `--reporter=json` per §3.3 and report PASS/FAIL per §3.4.
3. In the report, state **"this is a starter smoke, so extend it with real cases"** (`npx playwright codegen <baseURL>`).

> The runner is `npx playwright test` (deterministic, SSOT). Use the session's playwright MCP only as an
> auxiliary path for exploration when cases are absent (it is not the regression SSOT). MCP reference:
> the "playwright MCP" section of [`references/web-playwright.md`](references/web-playwright.md).

### 3.3 Deterministic Run

If cases exist, run deterministically with `--reporter=json` (+junit). `--reporter=json` emits to
**stdout** by default, so to capture it to a file, set `PLAYWRIGHT_JSON_OUTPUT_NAME` (otherwise the
file parsing in §3.4 breaks with ENOENT):

```bash
PLAYWRIGHT_JSON_OUTPUT_NAME=results.json \
PLAYWRIGHT_JUNIT_OUTPUT_NAME=results.xml \
  npx playwright test --reporter=json,junit
```

### 3.4 Parse Results and Report PASS/FAIL

Parse the `results.json` specified in §3.3:

```bash
# Result summary: extract pass/fail/skip counts
node -e "
  const r = JSON.parse(require('fs').readFileSync('results.json','utf8'));
  const s = r.stats;
  console.log('PASS:', s.expected, '/ FAIL:', s.unexpected, '/ SKIP:', s.skipped);
  process.exit(s.unexpected > 0 ? 1 : 0);
"
```

**Report format**:

```
## Integration Test Results — <date>

### Playwright Run Results
| Suite | Passed | Failed | Skipped | Verdict |
|---|---|---|---|---|
| tests/auth.spec.ts | 5 | 0 | 0 | PASS |
| tests/checkout.spec.ts | 3 | 1 | 0 | FAIL |

**Overall verdict**: FAIL (1 failure)
Failed case: tests/checkout.spec.ts › navigate to the confirmation page after checkout completes
```

## 4. If Non-Web — human-in-the-loop

> For per-type non-web signals and reference OSS details, see → [`references/non-web.md`](references/non-web.md).

Do not enforce automated integration testing. Collect scenarios and pass criteria via `AskUserQuestion`:

```
This project was detected as non-web (CLI/RN/Flutter, etc.).
No integration-test automation tool is enforced.

Please provide the following:
1. The core scenarios to verify (e.g., "look up data after user login")
2. The pass criteria for each scenario (e.g., "response code 200, data included")
3. Any test tool you are currently using, if any.
```

Based on the collected scenarios, write a manual verification checklist and point to applicable OSS
(Newman/Maestro/Appium — all Apache-2.0, not automatically enforced). For details, see
[`references/non-web.md`](references/non-web.md).
